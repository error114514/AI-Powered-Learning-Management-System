# 功能设计：教师端人脸签到专用逻辑

## 目的
在教师端新增一个专用的人脸签到入口，隐藏原始人脸签到和普通签到按钮；教师上传图片或视频进行比对，自动将匹配的学生加入课程签到表并给出清单提示。

**适用场景：** 前端已根据用户身份判断角色，若角色为学生则保持原有签到逻辑；若角色为教师，则启用以下新逻辑。

---

## 1) 用户界面行为（前端）

### 学生端
- 维持现有的普通签到和人脸签到按钮，按钮显示保持不变。

### 教师端
- 隐藏原有的"普通签到"和"人脸签到"按钮。
- 显示新的"教师人脸识别签到"按钮（按钮仅对教师可见）。
- 点击按钮后，弹出上传控件，允许上传图片或视频。
- 上传完成后，显示结果弹窗：
  - 如果找到符合条件的学生，弹出包含所有签入学生姓名的列表，并确认签入成功。
  - 如果未找到符合条件的学生，弹出提示窗"未找到符合条件的学生"。

---

## 2) 后端流程（简要）

### 入口 API
POST /api/courses/{course_id}/teacher_face_sign_in

**请求体包含：**
- 上传的图片或视频文件
- 用户身份：教师（服务器端鉴权）

### 处理流程
1. 从上传文件中提取图片帧（若为视频，分解为多帧）
2. 通过人脸识别模型将每帧提取的人脸嵌入向量
3. 与课程所属学生的已知脸库进行比对（逐个学生嵌入向量计算距离）
4. 设定距离阈值，筛选出"匹配学生"列表
5. 将匹配的学生逐条在课程签到表中创建签到记录
6. 组织返回数据：匹配到的学生姓名列表及其签到状态

### 响应
- **成功：** 返回 `{ success: true, matched_students: [ { id, name }, ... ] }`
- **未匹配：** 返回 `{ success: false, message: "未找到符合条件的学生" }`

---

## 3) 数据模型变更（简要）

### 新增/修改表

#### Attendance
- `id`
- `course_id`
- `student_id`
- `timestamp`
- `method`
- `source_image_id`
- `status`

`method` 可以标注为 `"teacher_face_sign_in"`

#### StudentImages 或 Embeddings
- `student_id`
- `image_path` 或 `embedding_vector`

用于人脸比对的特征数据

### 保留原有表
- Courses、Enrollments、Users、等（不变）

### 注意
- 尽量存储经过处理的特征（如嵌入向量），避免直接存储原始图片以提升隐私与安全性

---

## 4) 算法实现思路（可替代方案）

### 方案 A：使用现有系统的人脸识别模型（如 FaceNet / ArcFace）进行嵌入比对
**优点：** 鲁棒性好、成熟  
**实现要点：** 对每个学生保存稳定的嵌入向量；上传图片/视频后提取嵌入并与库中向量计算距离

### 方案 B：简单实现（便捷但鲁棒性较低）
**方案：** 对上传的图像/视频进行基础人脸检测，截取人脸区域后用简化特征（如更粗糙的对比）进行快速筛选  
**注意：** 不推荐在正式环境中使用低鲁棒性方案，但可作为快速原型

### 方案 C：多模态或分步比对
先筛选出候选学生集，再对候选集进行精确比对以提升速度

---

## 5) 安全与隐私要点

1. **权限控制：** 仅教师角色可使用该入口，学生端不可见
2. **数据隐私：** 尽量使用嵌入向量等特征数据代替原始照片存储
3. **数据最小化：** 只在需要时加载/比对相关学生数据
4. **日志审计：** 记录谁在何时进行了人脸签到操作
5. **用户知情：** 在系统隐私条款或使用说明中明确人脸数据的用途
